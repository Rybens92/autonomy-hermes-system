🔄 Autonomy Hermes System AI Agent Installation Guide for Multi-Profile Autonomous Pipelines

Version v1.0
Profiles 11 specialized
Cron Jobs 6 active (+2 disabled)
Scripts 6 active (+2 disabled)
For Hermes / Claude Code / Codex / OpenCode

Target audience: AI agents that need to replicate this autonomous agent system on a user's machine.
Goal: Install 11 specialized profiles, 6 active cron jobs (+2 disabled), 8 scripts, and a state directory — without breaking anything the user already has.

This guide is a decision tree, not a recipe. At every step, you MUST discover what the user already has and adapt. Never assume. Never overwrite.
WHEN IN DOUBT, ASK THE USER
This is the most important rule in this guide. You are not expected to guess.

Ask the user when:
  • You can't determine which model/provider to use for a tier
  • A profile name conflicts and you need an alternative
  • You're unsure whether a feature matches what the user wants
  • The user's setup is unconventional (e.g., custom providers, unusual paths)
  • You find existing cron jobs that might conflict
  • Any decision has multiple reasonable options with different trade-offs
How to ask (be specific):
  • ✗ "What should I do?"
  • ✓ "I need to choose between X and Y for the orchestrator model. X = faster but dumber. Y = slower but smarter. Which do you prefer?"
  • Always present concrete alternatives with trade-offs explained

0 Phase 0 — Discovery

Before creating anything, discover the user's environment. This determines every subsequent decision.

0.1 Profile discovery

# Expected output: list of profile names
# Store this list — you will check against it for naming conflicts
IF any profile name starts with "aa-" (e.g., aa-orchestrator, aa-worker): → This system may already be partially installed. → Ask user: "I see aa-* profiles. Are these from a previous installation? Should I overwrite or skip?" → If skip: create profiles with alternative prefix (e.g., "a2-") → If overwrite: proceed, but warn user

0.2 Provider and model discovery

hermes config list
# Identify: provider names, available models, API modes
#
# Key: which provider has the STRONGEST reasoning model?
# Key: which provider has the CHEAPEST capable model?
# These map to:
#   STRONGEST → aa-orchestrator, aa-researcher, aa-planner, aa-plan-taste, aa-thinker
#   CHEAPEST  → aa-worker, aa-explorer
#   BALANCED  → aa-reviewer, aa-human-taste, aa-verifier, aa-goalkeeper
Provider mapping logic — discover, don't assume:

The project creator uses the STRONGEST tier model for orchestration/review/taste and the CHEAPEST tier model for worker/explorer execution. But your user may have different providers configured — the system auto-maps.

How to choose:

  1. Run hermes tools list to see configured providers and models
  2. STRONGEST tier (orchestrator, researcher, planner, plan-taste, thinker): pick the provider with the best reasoning model
  3. CHEAPEST tier (worker, explorer): pick the provider with the cheapest capable model
  4. BALANCED tier (reviewer, taste, verifier, goalkeeper): pick a mid-tier model
  5. IF you can't determine which is best → ASK the user: "I need to choose models for the autonomous system. Your options: STRONGEST (orchestrator): [list providers with good reasoning] CHEAPEST (worker): [list providers with cheap capable models] BALANCED (review/taste): [list remaining options] Which provider/model for each tier?"
  6. IF only ONE provider is configured → use it for all tiers, picking the strongest available model for STRONGEST, cheapest for CHEAPEST

cat > {HOME}/autonomous-agent/GOAL.md << 'EOF'
# Your Goal

Write what you want the autonomous agent to work on. Be specific but leave
implementation details to the system.

Repository: /path/to/your/project

## Tasks
1. First task
2. Second task
...

## Rules
- Code and documentation in English
- Every change goes through pipeline: worker → reviewer → aa-human-taste → verifier
- If reviewer or verifier rejects → fix and re-verify
- Do not modify Hermes configuration files

# PROJECT.md — project path (one line)
echo "/path/to/your/project" > {HOME}/autonomous-agent/PROJECT.md

agent:
  max_turns: 30
  reasoning_effort: high

    You are the central orchestrator of an autonomous agent system.
    Your profile is aa-orchestrator. You operate primarily as a
    cron-driven background agent, but also handle direct user
    communication via Telegram. You do NOT write code or use the
    terminal.

    Your tools: file (reading/writing state files: GOAL.md, STATUS.md,
    LOG.md, HELP_NEEDED.md) and kanban (delegating work by creating
    cards with assignee). You do NOT have delegate_task. All execution
    goes through kanban to worker profiles.

    ═══ DUAL ROLE ═══

    1. CRON MODE — Autonomous orchestration (hourly heartbeat):
       Execute goals from GOAL.md by creating kanban cards, monitoring
       progress, performing continuous audit. Work autonomously — do
       NOT wait for user confirmation.

    2. CHAT MODE — Direct user communication (Telegram):
       When the user sends a message, respond directly. Read system
       files to answer questions about status. When the user asks to
       change goals, priorities, or settings, write to the appropriate
       state files. Be concise and actionable.

    ═══ IDLE GUARD + CONTINUOUS AUDIT ═══

    The system NEVER goes truly idle — when all active work is done,
    it continuously audits and improves completed work forever.

    ═══ GOAL FRESHNESS TRACKING ═══

    STATUS.md MUST maintain a Goal Tracker section with GOAL.md hash
    and per-objective status (DONE / IN PROGRESS / NOT STARTED),
    with card IDs for each.

    ═══ DECISION TREE ═══

    0. GOAL.md FRESHNESS CHECK — RUN THIS FIRST, ALWAYS.
       a. Read GOAL.md objectives.
       b. Read Goal Tracker in STATUS.md.
       c. For EACH objective: does it have ≥1 kanban card?
       d. IF any objective has ZERO cards → CREATE KANBAN CARDS
          for it RIGHT NOW. Skip audit. New goals come first.
       e. IF all objectives have cards → proceed.

    1. Board has ready/failed WORK tasks? → act, skip audit.
    2. Board empty + audit work remaining → run CONTINUOUS AUDIT.
    3. Board empty + audit cycle complete → increment Cycle, reset
       cursor, start next cycle. Perpetual improvement.

    ═══ AUDIT CURSOR ═══

    STATUS.md tracks: Cycle number, total done tasks, last audited
    task ID, audited count, audit depth (cycle-specific focus).
    One task per tick. Rotate through all done tasks infinitely.

    CYCLE 1 — VERIFICATION: Do deliverables still meet requirements?
    CYCLE 2 — QUALITY: Can output be substantially better?
    CYCLE 3+ — DEEP IMPROVEMENT: Edge cases, synergies, alternatives

    ═══ TASTE-ESCALATOR (inline — no separate profile) ═══

    Before delegating sensitive work, evaluate thresholds:
    - impact ≤ 2 OR sim_confidence ≥ 4 → DELEGATE (autonomous)
    - impact ≥ 4 AND reversibility ≤ 2 → flag HELP_NEEDED.md
    - impact ≥ 4 AND sim_confidence ≤ 2 → flag HELP_NEEDED.md
    - impact ≥ 3 AND reversibility ≤ 2 → flag HELP_NEEDED.md
    - ELSE → DELEGATE

    When flagging HELP_NEEDED.md: write specific decision point,
    alternatives, trade-offs, recommendation. Continue other work —
    do NOT block on resolution.

    ═══ WORKFLOW ═══

    0. PRE-FLIGHT: verify workspace_path exists before routing.
    1. GOAL SCANNING: new objectives → create worker/thinker card.
    2. ROUTING: determine pre-worker stages (explorer, researcher,
       planner, plan-taste, or skip to worker/thinker).
    3. PLANNING GATE: plan-taste APPROVE/REVISE/REJECT.
    4. PIPELINE CHAINING: create full chain in one tick using
       parent dependencies (explorer → planner → plan-taste →
       worker → reviewer → taste → verifier → goalkeeper).
    5. Evaluate via taste-escalator → escalate to user or delegate.
    6. Update STATUS.md (write_file).
    7. Append LOG.md: [TIMESTAMP] [DECISION] description.
    8. Blocked → HELP_NEEDED.md and end.

    ═══ PROFILE ASSIGNMENT GUIDE ═══

    - aa-explorer: filesystem discovery (read-only)
    - aa-researcher: web research with citations
    - aa-planner: task decomposition (never executes)
    - aa-plan-taste: plan verification (APPROVE/REVISE/REJECT)
    - aa-worker: code, terminal, research execution
    - aa-reviewer: technical quality (0-10, approve/reject)
    - aa-human-taste: AI-slop detection
    - aa-verifier: hallucinations, facts, adversarial check
    - aa-thinker: architecture, deep reasoning, hard problems
    - aa-goalkeeper: global goal achievement assessment

    ═══ RULES ═══
    - Tools are your ONLY action. Call tools, don't describe plans.
    - One decision per cron tick.
    - Every task pipeline: explorer [+ researcher] → planner →
      plan-taste → worker → reviewer → taste → verifier → goalkeeper.
      Simple tasks skip pre-worker stages.
    - Card body must be concrete and precise with acceptance criteria.

3.1 state_context.sh

Purpose: Injected as context before every orchestrator tick. Reads all state files and the kanban board. Has HELP_NEEDED guard — if help is needed, outputs minimal prompt to save tokens.

#!/bin/bash
# state_context.sh — injects state + prompt into the orchestrator agent
# Cron job calls this without arguments — everything is in stdout

BASE="{HOME}/autonomous-agent"
HELP_FILE="$BASE/HELP_NEEDED.md"
PROJECT_FILE="$BASE/PROJECT.md"

# ── HELP_NEEDED guard: if blocked, send TINY prompt to save tokens ──
if [ -s "$HELP_FILE" ]; then
  echo "Blocker detected: HELP_NEEDED.md is non-empty."
  echo "Do nothing this tick. No tool calls. No file writes. Exit immediately."
  echo "Waiting for user response. Token saving is priority."
  exit 0
fi

# ── Normal context: full state injection ──
echo "=== GOAL ==="
cat "$BASE/GOAL.md" 2>/dev/null || echo "(no goal)"
echo ""
echo "=== STATUS ==="
cat "$BASE/STATUS.md" 2>/dev/null || echo "(no status)"
echo ""
echo "=== LOG (last 20 lines) ==="
tail -20 "$BASE/LOG.md" 2>/dev/null || echo "(no log)"
echo ""
echo "=== PROJECT PATH ==="
cat "$PROJECT_FILE" 2>/dev/null || echo "{HOME}/workspace"
echo ""
echo "=== HELP SIGNAL ==="
cat "$HELP_FILE" 2>/dev/null || echo "(none)"
echo ""
echo "=== KANBAN BOARD ==="

# Use full path to hermes — cron jobs may not have it on PATH
HERMES_BIN="{HERMES_INSTALL}/venv/bin/hermes"
$HERMES_BIN -p aa-orchestrator kanban list 2>/dev/null || echo "(kanban unavailable)"
echo ""
echo "=== CRON STATUS ==="
$HERMES_BIN -p aa-orchestrator cron list 2>/dev/null || echo "(cron unavailable)"
echo ""
echo "=== END CONTEXT ==="
echo ""

3.4 cleanup.sh

Purpose: Periodic maintenance — archive done tasks >24h, alert on blocked tasks >12h, remove empty observations and stale temp files. No-agent.

#!/bin/bash
# cleanup.sh — periodic maintenance for the autonomous system
# no-agent job: stdout delivered verbatim. Empty = silent (nothing to clean).

HERMES_BIN="{HERMES_INSTALL}/venv/bin/hermes"
BASE="{HOME}/autonomous-agent"
CHANGES=0

# Archive done tasks older than 24h
$HERMES_BIN -p aa-orchestrator kanban gc --older-than 24h 2>/dev/null && CHANGES=1

# Alert on blocked tasks older than 12h
BLOCKED=$($HERMES_BIN -p aa-orchestrator kanban list 2>/dev/null | grep -c "blocked")
if [ "$BLOCKED" -gt 0 ]; then
  echo "⚠️  $BLOCKED task(s) blocked for >12h — check kanban board"
  CHANGES=1
fi

# Remove empty observation files >24h old
find "$BASE/observations" -name "*.md" -size 0 -mtime +1 -delete 2>/dev/null

# Clean stale temp files
find /tmp -name "hermes_*" -mtime +1 -delete 2>/dev/null

# Only output if we did something
[ "$CHANGES" -eq 0 ] && exit 0
echo "🧹 Cleanup completed at $(date)"

Troubleshooting

Orchestrator creates tasks but nothing happens

  • Check: hermes -p aa-orchestrator kanban list — are tasks in "ready"?
  • Check: systemctl --user status hermes-gateway-aa-orchestrator — is the dispatcher running?
  • The dispatcher claims and runs "ready" tasks every 60 seconds. If tasks stay "ready" >2 minutes, the gateway dispatcher is stuck.

Workers crash with "protocol violation"

  • Worker exited without calling kanban_complete or kanban_block
  • Fix: ensure every worker profile has kanban_complete in its system prompt
  • Check: grep kanban_complete {HOME}/.hermes/profiles/aa-*/config.yaml

HELP_NEEDED messages not arriving

  • Check: cat {HOME}/autonomous-agent/HELP_NEEDED.md — is it non-empty?
  • Check: hermes -p aa-orchestrator cron log [watchdog-job-id] — any errors?
  • Check: hermes channels — is messaging platform configured?
  • Check backup: ls {HOME}/autonomous-agent/HELP_NEEDED.md.*.sent — if backup exists but you never got the message, delivery failed silently

Too many tokens consumed

  • Reduce heartbeat frequency: hermes -p aa-orchestrator cron edit [id] --schedule "0 */2 * * *" (every 2h)
  • Reduce observer frequency: hermes -p aa-orchestrator cron edit [id] --schedule "0 * * * *" (hourly)
  • The idle guard already saves tokens when nothing to do
  • Verify: cat {HOME}/autonomous-agent/LOG.md — how many active vs idle ticks?

Adaptation Guidelines for AI Agents

GOLDEN RULE: This guide describes the IDEAL system. The user's environment WILL differ. Your job is to ADAPT, not just copy. Every {PLACEHOLDER} must be resolved from discovery data, not assumed.

Key adaptation points

  1. Provider/model names — never hardcode. Map from discovery.
  2. Profile prefix — default "aa-", but check for conflicts.
  3. Message platform — Telegram, Discord, Slack, etc. Detect from channels.
  4. Path to hermes binary — resolve from installation. Try: which hermes, ~/.hermes/hermes-agent/venv/bin/hermes, ~/.local/bin/hermes.
  5. Home directory — $HOME always. Never hardcode /home/username.
  6. Working directory — $HOME/workspace if it exists, otherwise $HOME.
  7. Max turns per profile — adjust based on user's model tier. Strong models can have more turns.
  8. Model choice for BALANCED tier — if user has only 2 providers, use the cheaper one for BALANCED to save tokens.

What to NEVER do

  • ✗ Overwrite an existing profile with the same name
  • ✗ Hardcode API keys or provider names
  • ✗ Modify the user's default profile
  • ✗ Change the user's existing cron jobs
  • ✗ Assume a specific messaging platform
  • ✗ Install without asking if profiles/cron already exist
