🔄 Autonomy Hermes System
AI Agent Installation Guide — v1.1

Target audience: AI agents (Hermes, Claude Code, Codex, OpenCode) that need to replicate this autonomous agent system on a user's machine.
Goal: Install 9 specialized profiles, 8 cron jobs, 9 scripts, and a state directory — without breaking anything the user already has.

This guide is a decision tree, not a recipe. At every step, you MUST discover what the user already has and adapt. Never assume. Never overwrite.
⚠️ WHEN IN DOUBT, ASK THE USER
This is the most important rule in this guide. You are not expected to guess.

Ask the user when:
  • You can't determine which model/provider to use for a tier
  • A profile name conflicts and you need an alternative
  • You're unsure whether a feature matches what the user wants
  • The user's setup is unconventional (e.g., custom providers, unusual paths)
  • You find existing cron jobs that might conflict
  • Any decision has multiple reasonable options with different trade-offs
How to ask (be specific):
  • ❌ "What should I do?"
  • ✅ "I need to choose between X and Y for the orchestrator model. X = faster but dumber. Y = slower but smarter. Which do you prefer?"
  • Always present concrete alternatives with trade-offs explained

Phase 0 — Discovery

Before creating anything, discover the user's environment. This determines every subsequent decision.

0.1 Profile discovery

Should I overwrite or skip?" → If skip: create profiles with alternative prefix (e.g., "a2-") → If overwrite: proceed, but warn user

0.2 Provider and model discovery

hermes config list
# Identify: provider names, available models, API modes
#
# Key: which provider has the STRONGEST reasoning model?
# Key: which provider has the CHEAPEST capable model?
# These map to:
#   STRONGEST → aa-orchestrator, aa-thinker
#   CHEAPEST  → aa-worker
#   BALANCED  → aa-reviewer, aa-human-taste, aa-verifier, aa-escalator, aa-goalkeeper
Provider mapping logic — discover, don't assume:

The project creator uses GLM-5.2 via z.ai for orchestration and DeepSeek v4 (Flash for workers, Pro for review/taste) for execution. But your user almost certainly has different providers configured.

How to choose:

  1. Run hermes tools list to see configured providers and models
  2. STRONGEST tier (orchestrator, thinker): pick the provider with the best reasoning model
  3. CHEAPEST tier (worker): pick the provider with the cheapest capable model
  4. BALANCED tier (reviewer, taste, verifier, escalator, goalkeeper): pick a mid-tier model
  5. IF you can't determine which is best → ASK the user: "I need to choose models for the autonomous system. Your options: STRONGEST (orchestrator): [list providers with good reasoning] CHEAPEST (worker): [list providers with cheap capable models] BALANCED (review/taste): [list remaining options] Which provider/model for each tier?"
  6. IF only ONE provider is configured → use it for all tiers, picking the strongest available model for STRONGEST, cheapest for CHEAPEST

2.3 Detailed prompts by profile

aa-orchestrator

provider: {PROVIDER_STRONG}
model: {MODEL_STRONG}
toolsets:
  - file
  - kanban
agent:
  max_turns: 30
  reasoning_effort: high
  system_prompt: |
    You are an autonomous orchestrator. You do NOT write code. You do NOT use the terminal.
    Do NOT describe what you INTEND to do — CALL the tool and DO it.
    Your tools: file (ONLY for reading/writing state files), kanban (creating tasks for workers).

    ═══ IDLE GUARD (check this first) ═══
    Before making any decision, check the kanban board state in context.
    If ALL tasks are "running" or "done" and there's nothing new to do —
    do NOT create cards. But do NOT just idle mindlessly.

    IDLE ≠ end of work. IDLE = reflection opportunity:
    1. If GOAL.md tasks are IN PROGRESS (board not empty) —
       update STATUS.md with "idle — waiting for workers" and finish.
    2. If ALL GOAL.md tasks are COMPLETE (board empty, all done):
       a. SELF-AUDIT: review LOG.md and STATUS.md, evaluate work quality
          from a human perspective. What would the user do differently?
       b. If improvements needed → create "polish" card with priority=0
          (non-blocking, worker fixes in idle time)
       c. If everything is well done → do NOT stay silent. ASK via
          HELP_NEEDED.md: "GOAL.md complete. What next? Based on your
          interests: [proposals from research/proposals/]. Should I work
          on any of these?"
       d. NEVER idle silently when all work is done — that wastes tokens
          and misses opportunities.

    ═══ HELP_NEEDED GUARD ═══
    If HELP_NEEDED.md is non-empty in context: do NOT create cards.
    Do NOT update STATUS.md or LOG.md. Just confirm you're waiting.
    Save tokens — user already knows about the blocker via messaging.
    Act ONLY if HELP_NEEDED content changed (user responded).

    ═══ TASTE-ESCALATOR ═══
    Before sensitive decisions (architecture changes, library choices,
    significant scope changes) — before delegating:
    1. Create a card for aa-escalator: evaluate impact(1-5),
       reversibility(1-5), sim_confidence(1-5).
       If impact≥3 and sim_confidence≤2 → ESCALATE to user.
       Otherwise → DELEGATE, proceed automatically.

    ═══ WORKFLOW ═══
    1. Check idle guard
    2. Check HELP_NEEDED guard
    3. If work exists: evaluate via aa-escalator whether user input needed
    4. Create kanban card with kanban_create, assigning to the right profile.
       CRITICAL: EVERY card MUST have:
       - workspace_kind: "dir"
       - workspace_path: the project path from PROJECT.md (injected in context)
       - assignee: profile name (NEVER omit — tasks without assignee never run)
       - body: full task description with acceptance criteria
       Profile to use:
       - aa-worker: task execution (code, terminal, research)
       - aa-reviewer: technical quality assessment (0-10, approve/reject)
       - aa-human-taste: human-like quality, AI-slop detection
       - aa-verifier: hallucinations, facts, adversarial check
       - aa-escalator: meta-decision whether real user input needed
       - aa-thinker: hard problems (architecture, deep reasoning)
       - aa-goalkeeper: global assessment whether goal achieved
    5. write_file: update STATUS.md (new state)
    6. write_file: append to LOG.md: [TIMESTAMP] [DECISION] description
    7. If completely blocked → write_file to HELP_NEEDED.md and exit

    ═══ RULES ═══
    - Tools are your ONLY action. Do not write "I will create a card" —
      CALL kanban_create. LOG.md description appears AFTER execution.
    - One decision per tick. Don't try to do everything at once.
    - If you don't know what to do → read LOG.md (read_file), don't guess.
    - Create precise cards with concrete task descriptions (body).
    - Every piece of work: worker → reviewer → aa-human-taste+verifier.
    - You are quadriplegic. Strength is in orchestration, not execution.

    ═══ HUMAN-TASTE EXECUTION ═══
    When working on ANY task from GOAL.md:
    1. What would a human expect from this system? Is the output useful,
       or just AI-slop nobody would want to read?
    2. When a decision is unclear or controversial (sim_confidence ≤ 2,
       new territory without precedent) → do NOT guess. ESCALATE to user
       via HELP_NEEDED.md. Ask a CONCRETE question with alternatives.
    3. When human taste simulation might fail (genuinely new feature,
       architecture change, choice between equivalent options) → ESCALATE.
    4. For routine, well-understood tasks (sim_confidence ≥ 4, impact ≤ 2)
       → act autonomously, don't ask.
    5. After escalation — WAIT for user response. Do NOT create cards or
       make decisions until HELP_NEEDED.md is cleared.

    ═══ SLOP-POLISH (after APPROVE with flags) ═══
    When aa-human-taste or reviewer APPROVES but flags specific issues —
    do NOT ignore them. After phase completion and APPROVE verification,
    create a "polish" card for aa-worker with priority=0 containing ALL
    unfixed flags as a checklist. This doesn't block the pipeline.

3.1 state_context.sh

Purpose: Injected as context before every orchestrator tick. Reads all state files and the kanban board. Has HELP_NEEDED guard — if help is needed, outputs minimal prompt to save tokens.

#!/bin/bash
# state_context.sh — injects state + prompt into the orchestrator agent
# Cron job calls this without arguments — everything is in stdout

BASE="{HOME}/autonomous-agent"
HELP_FILE="$BASE/HELP_NEEDED.md"
PROJECT_FILE="$BASE/PROJECT.md"

# ── HELP_NEEDED guard: if blocked, send TINY prompt to save tokens ──
if [ -s "$HELP_FILE" ]; then
  echo "Blocker detected: HELP_NEEDED.md is non-empty."
  echo "Do nothing this tick. No tool calls. No file writes. Exit immediately."
  echo "Waiting for user response. Token saving is priority."
  exit 0
fi

# ── Normal context: full state injection ──
echo "=== GOAL ==="
cat "$BASE/GOAL.md" 2>/dev/null || echo "(no goal)"
echo ""
echo "=== STATUS ==="
cat "$BASE/STATUS.md" 2>/dev/null || echo "(no status)"
echo ""
echo "=== LOG (last 20 lines) ==="
tail -20 "$BASE/LOG.md" 2>/dev/null || echo "(no log)"
echo ""
echo "=== PROJECT PATH ==="
cat "$PROJECT_FILE" 2>/dev/null || echo "{HOME}/workspace"
echo ""
echo "=== HELP SIGNAL ==="
cat "$HELP_FILE" 2>/dev/null || echo "(none)"
echo ""
echo "=== KANBAN BOARD ==="

# Use full path to hermes — cron jobs may not have it on PATH
HERMES_BIN="{HERMES_INSTALL}/venv/bin/hermes"
$HERMES_BIN -p aa-orchestrator kanban list 2>/dev/null || echo "(kanban unavailable)"
echo ""
echo "=== CRON STATUS ==="
$HERMES_BIN -p aa-orchestrator cron list 2>/dev/null || echo "(cron unavailable)"
echo ""
echo "=== END CONTEXT ==="
echo ""
echo "TASK: Evaluate current state and make ONE decision. Follow the rules in your system prompt."
echo "If idle: self-audit completed work. If all done: ask user for direction via HELP_NEEDED."
echo "If board has ready/failed tasks: act on them."
echo "If board has running tasks and nothing new: mark idle and exit."
echo "Create card with: workspace_kind=dir, workspace_path=$(cat $PROJECT_FILE 2>/dev/null || echo '{HOME}/workspace')"

3.2 check_help.sh

Purpose: Watchdog — monitors HELP_NEEDED.md and delivers to user's messaging platform. No-agent: stdout is delivered verbatim. Silent when no help needed.

#!/bin/bash
# check_help.sh — watchdog: if HELP_NEEDED.md is non-empty, deliver to messaging
# no-agent job: stdout delivered verbatim. Empty stdout = silent.
HELP_FILE="{HOME}/autonomous-agent/HELP_NEEDED.md"
if [ -s "$HELP_FILE" ]; then
  # Backup BEFORE delivery attempt — if delivery fails, help request survives
  BACKUP="${{HELP_FILE}}.{{$(date +%s)}}.sent"
  cp "$HELP_FILE" "$BACKUP"

  echo "🚨 AUTONOMOUS AGENT NEEDS HELP:"
  echo "---"
  cat "$HELP_FILE"
  echo "---"
  echo "Timestamp: $(date '+%Y-%m-%d %H:%M:%S')"

  # Only clear after successful output. Backup preserved for audit trail.
  true > "$HELP_FILE"
fi

3.3 milestone_reporter.sh

Purpose: Detects STATUS.md changes and reports phase transitions to user. Deduplicates via hash comparison. No-agent.

#!/bin/bash
# milestone_reporter.sh — report status changes to user's messaging
# no-agent job: stdout delivered verbatim. Empty = silent (no change).

STATUS_FILE="{HOME}/autonomous-agent/STATUS.md"
HASH_FILE="{HOME}/.hermes/profiles/aa-orchestrator/scripts/.milestone_hash"

  echo "---"
  echo "$NEW_HASH" > "$HASH_FILE"
fi

3.4 cleanup.sh

Purpose: Periodic maintenance — archive done tasks >24h, alert on blocked tasks >12h, remove empty observations and stale temp files. No-agent.

#!/bin/bash
# cleanup.sh — periodic maintenance for the autonomous system
# no-agent job: stdout delivered verbatim. Empty = silent (nothing to clean).

HERMES_BIN="{HERMES_INSTALL}/venv/bin/hermes"
BASE="{HOME}/autonomous-agent"
CHANGES=0

# Archive done tasks older than 24h
$HERMES_BIN -p aa-orchestrator kanban gc --older-than 24h 2>/dev/null && CHANGES=1

# Alert on blocked tasks older than 12h
BLOCKED=$($HERMES_BIN -p aa-orchestrator kanban list 2>/dev/null | grep -c "blocked")
if [ "$BLOCKED" -gt 0 ]; then
  echo "⚠️  $BLOCKED task(s) blocked for >12h — check kanban board"
  CHANGES=1
fi

# Remove empty observation files >24h old
find "$BASE/observations" -name "*.md" -size 0 -mtime +1 -delete 2>/dev/null

# Clean stale temp files
find /tmp -name "hermes_*" -mtime +1 -delete 2>/dev/null

# Only output if we did something
[ "$CHANGES" -eq 0 ] && exit 0
echo "🧹 Cleanup completed at $(date)"

Troubleshooting

Orchestrator creates tasks but nothing happens

  • Check: hermes -p aa-orchestrator kanban list — are tasks in "ready"?
  • Check: systemctl --user status hermes-gateway-aa-orchestrator — is the dispatcher running?
  • The dispatcher claims and runs "ready" tasks every 60 seconds. If tasks stay "ready" >2 minutes, the gateway dispatcher is stuck.

Workers crash with "protocol violation"

  • Worker exited without calling kanban_complete or kanban_block
  • Fix: ensure every worker profile has kanban_complete in its system prompt
  • Check: grep kanban_complete {HOME}/.hermes/profiles/aa-*/config.yaml

HELP_NEEDED messages not arriving

  • Check: cat {HOME}/autonomous-agent/HELP_NEEDED.md — is it non-empty?
  • Check: hermes -p aa-orchestrator cron log [watchdog-job-id] — any errors?
  • Check: hermes channels — is messaging platform configured?
  • Check backup: ls {HOME}/autonomous-agent/HELP_NEEDED.md.*.sent — if backup exists but you never got the message, delivery failed silently

Too many tokens consumed

  • Reduce heartbeat frequency: hermes -p aa-orchestrator cron edit [id] --schedule "0 */2 * * *" (every 2h)
  • Reduce observer frequency: hermes -p aa-orchestrator cron edit [id] --schedule "0 * * * *" (hourly)
  • The idle guard already saves tokens when nothing to do
  • Verify: cat {HOME}/autonomous-agent/LOG.md — how many active vs idle ticks?

Adaptation Guidelines for AI Agents

GOLDEN RULE: This guide describes the IDEAL system. The user's environment WILL differ. Your job is to ADAPT, not just copy. Every {PLACEHOLDER} must be resolved from discovery data, not assumed.

Key adaptation points

  1. Provider/model names — never hardcode. Map from discovery.
  2. Profile prefix — default "aa-", but check for conflicts.
  3. Message platform — Telegram, Discord, Slack, etc. Detect from channels.
  4. Path to hermes binary — resolve from installation. Try: which hermes, ~/.hermes/hermes-agent/venv/bin/hermes, ~/.local/bin/hermes.
  5. Home directory — $HOME always. Never hardcode /home/username.
  6. Working directory — $HOME/workspace if it exists, otherwise $HOME.
  7. Max turns per profile — adjust based on user's model tier. Strong models can have more turns.
  8. Model choice for BALANCED tier — if user has only 2 providers, use the cheaper one for BALANCED to save tokens. Note: aa-comms uses BALANCED tier but only needs the file toolset — it has no kanban, terminal, or web access, and is intentionally the least privileged profile.

What to NEVER do

  • ✗ Overwrite an existing profile with the same name
  • ✗ Hardcode API keys or provider names
  • ✗ Modify the user's default profile
  • ✗ Change the user's existing cron jobs
  • ✗ Assume a specific messaging platform
  • ✗ Install without asking if profiles/cron already exist
